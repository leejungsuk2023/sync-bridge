# BBG Automation Roadmap
> 작성일: 2026년 3월 | 목적: 운영 자동화 로드맵

---

## 1. 전체 방향성

### 배경
BBG는 한국 병원을 한국 거주 태국인에게 마케팅하는 대행사로, SyncBridge를 핵심 운영 허브로 삼아 주요 업무를 단계적으로 자동화한다.

### 완료된 자동화
| 항목 | 내용 |
|------|------|
| AI 캡션 지침 | 금지단어 리스트 + 병원별 시술 지침 기반 AI 캡션 생성 |
| 디자인 텍스트 제거 | 이미지 내 텍스트 제거 규칙 적용 → 재작업 핑퐁 제거 |

## 3. 1순위: 태국 Sales 성과 트래킹

### 왜 해야 하는가
현재 태국 Sales의 응대 품질과 전환율을 파악할 방법이 없다. 젠데스크에 대화 내용은 있지만 성과 트래킹이 전혀 없는 상태다. 실제로 986건의 문의 중 **520건(53%)이 미배정 상태**로 방치되고 있어 잠재 고객 손실이 크다.

> 메타데이터(응답시간, 처리건수)만으로는 "잘 응대했는지" 알 수 없다. 대화 내용을 봐야 예약 연결 여부, 응대 품질을 판단할 수 있기 때문에 **대화 내용까지 반드시 가져와야 한다.**

### 어떻게 할 것인가

**데이터 소스**
- 젠데스크 API → 티켓별 메타데이터 + **대화 내용(comments) 전체** 수집
- 티켓 1건씩 순차적으로 Gemini 2.5 Pro에 넘겨 분석 (한꺼번에 처리 X)
- 분석 결과만 Supabase에 저장 → 누적 데이터로 성과 트래킹

**토큰 비용 관리**
- 티켓 전체를 한번에 넣지 않고 **1건씩 순차 처리**
- 티켓 1건 평균 대화 10~20줄 기준 → 비용 미미한 수준
- 신규 티켓만 증분 처리 (이미 분석한 티켓은 재처리 안 함)
- 월 전체 비용 수천 원 ~ 수만 원 수준으로 관리 가능

**수집 및 분석 지표**
| 지표 | 수집 방법 | 설명 |
|------|----------|------|
| 첫 응답 시간 | 젠데스크 API 메타데이터 | 문의 접수 후 첫 응답까지 걸린 시간 |
| 처리 건수 | 젠데스크 API 메타데이터 | 담당자별 일/주/월 티켓 처리 수 |
| 미응답 건수 | 젠데스크 API 메타데이터 | 배정됐지만 응답 없는 티켓 수 |
| 응대 품질 | Gemini 대화 분석 | 친절도, 정보 정확성, 문제 해결 여부 |
| 예약 전환 여부 | Gemini 대화 분석 | 대화 내용에서 예약 성사 여부 판단 |
| 팔로업 필요 여부 | Gemini 대화 분석 | 재상담 필요한 잠재 고객 자동 추출 |

**구현 흐름**
```
젠데스크 API
→ 티켓 메타데이터 + 대화 내용 수집
→ 티켓 1건씩 Gemini 2.5 Pro 분석
→ 결과 Supabase 저장
→ SyncBridge God Mode 대시보드에 Sales 성과 섹션 표시
```

**기대 효과**
- Sales별 응대 품질과 전환율이 투명하게 보임
- 응대 품질 + 전환율까지 정량적으로 측정 가능
- 미배정 520건 문제 즉시 가시화
- 팔로업 필요 고객 자동 추출 (3순위와 연결)

## 4. 2순위: 젠데스크 + 메타 이상 알림

### 왜 해야 하는가
현재 관리팀이 하루에도 여러 번 젠데스크와 메타 광고 관리자를 수동으로 들여다보며 이상 징후를 찾고 있다. 이 수시 점검 업무가 관리팀 시간의 상당 부분을 차지한다. 자동화되면 관리팀의 점검 업무가 사라진다.

### 어떻게 할 것인가

**감지 조건**
| 조건 | 알림 내용 |
|------|----------|
| 병원별 문의 3일 이상 0건 | "XX병원 3일째 문의 없음. 광고 점검 필요" |
| 광고비 소진 중 전환 없음 | "XX병원 광고비 집행 중이나 문의 0건" |
| 메타 광고 갑작스러운 중단 | "XX병원 광고 중단 감지" |
| 미응답 문의 24시간 초과 | "XX병원 미응답 문의 N건" |

**구현 방식**
- 젠데스크 API + 메타 광고 API 데이터 주기적 수집 (1시간 단위)
- Supabase Edge Function + pg_cron으로 스케줄링
- SyncBridge God Mode SLA 신호등 섹션에 병원별 이상 감지 추가
- 이상 감지 시 SyncBridge 내 알림 표시

**기대 효과**
- 관리팀 수시 점검 업무 제거
- 문제 병원 즉시 감지 → 이탈 방어 가능

## 5. 3순위: 팔로업 대상 자동 추출

### 왜 해야 하는가
현재 문의가 와도 팔로업이 체계적으로 되지 않고 있다. 상담은 했지만 예약으로 연결 안 된 잠재 고객들이 방치되고 있으며, 이걸 자동으로 뽑아서 Sales에게 배정하면 전환율이 올라간다.

### 어떻게 할 것인가

**팔로업 대상 기준**
| 조건 | 설명 |
|------|------|
| 상담 후 X일 내 예약 없음 | 관심은 있지만 아직 미결정 고객 |
| 이전 방문 후 재방문 없음 | 재유치 가능성 있는 고객 |
| 문의 후 미응답 | Sales가 응답 못한 잠재 고객 |

**구현 방식**
- 젠데스크 API로 팔로업 대상 자동 추출
- SyncBridge Worker Extension에 팔로업 대상 리스트 자동 표시
- Sales가 팔로업 결과를 SyncBridge에 직접 입력
- Gemini 2.5 Pro로 고객별 맞춤 팔로업 메시지 초안 자동 생성 (태국어)

**기대 효과**
- 방치되는 잠재 고객 제로화
- 문의 → 예약 전환율 향상
- Sales 팔로업 업무 체계화 → 성과 측정 가능

## 6. 4순위: 병원 프로모션 취합 자동화

### 왜 해야 하는가
베트남 마케터가 15개 병원의 프로모션 정보를 매달 수동으로 수집하고 인스타/페북에 포스팅하고 있다. 수집 과정이 비효율적이며 이를 자동화하면 업무 부담이 크게 줄어든다.

### 어떻게 할 것인가
- 매달 초 병원별 구글 폼으로 프로모션 정보 자동 요청
- 취합된 데이터 자동 정리 → 베트남 마케터 검토 후 승인
- 메타 예약 포스팅으로 수동 포스팅 제거

## 7. 기술 스택

| 구분 | 기술 |
|------|------|
| 내부 플랫폼 | SyncBridge (Next.js + Supabase + Electron) |
| CS 데이터 | 젠데스크 API |
| 광고 데이터 | 메타 광고 API |
| AI 분석 | Gemini 2.5 Pro |
| 알림 | SyncBridge 내부 알림 시스템 |
| DB | Supabase (PostgreSQL + Realtime) |
| 스케줄러 | Supabase Edge Function + pg_cron |
